# HekaDrop Protocol — Folder Payload (`HEKABUND` v1, wire-level spec)

- **Protocol family:** HekaDrop extensions to Google Quick Share wire format
- **Version:** `folder_stream_v1` (capabilities bit `0x0004`)
- **Status:** Draft, hedef implementation v0.8.0
- **Normative RFC:** [`docs/rfcs/0005-folder-payload.md`](../rfcs/0005-folder-payload.md)
- **Related:** [`capabilities.md`](capabilities.md) — `HekaDropFrame` envelope and
  capability negotiation; [`chunk-hmac.md`](chunk-hmac.md) — RFC-0003 chunk
  integrity that wraps the bundle on the wire; [`resume.md`](resume.md) —
  RFC-0004 resume semantics carry through unchanged at byte level.
- **Audience:** protocol implementers (Android router authors, 3rd party Quick
  Share clients, fuzz harness writers, audit vendors)

This document is the **byte-exact reference** for HekaDrop's folder transfer
extension. The prose, motivation, alternatives, and trade-offs live in
RFC-0005. If the two documents disagree, the RFC is authoritative for design
intent and this document is authoritative for bytes on the wire.

---

## 1. Why this exists (one paragraph)

Quick Share's wire format addresses a single file per `FileMetadata`. Drag-
dropping a folder today flattens it: N files arrive in `~/Downloads/` with
their parent directory structure lost (RFC-0005 §1, §2). NearDrop has had
the same gap open for two years (issue #211); LocalSend issues an HTTP
request per file with no folder grouping primitive. To preserve directory
structure, atomic delivery, and bundle-level integrity *without* modifying
upstream proto schemas, HekaDrop wraps a folder into a single custom binary
container — `HEKABUND` v1 — that traverses the wire as one ordinary `FILE`
payload with a distinguishing MIME marker. Receivers that advertise the
`FOLDER_STREAM_V1` capability (`0x0004`) extract the bundle to a temporary
directory, verify per-file SHA-256 against the in-bundle manifest, and
atomic-rename the directory into `~/Downloads/` on full success. Any
integrity failure discards the entire bundle (atomic-reject; RFC-0005 §3.6).

---

## 2. Wire layout — `HEKABUND` v1 container

The bundle is a *self-contained* binary blob: it is generated by the sender
in memory (no on-disk staging; RFC-0005 alternative E rejected), streamed
through the existing `PayloadTransferFrame` chunk pipeline, and reconstructed
by the receiver into a `~/Downloads/.hekadrop-temp-<session>.bundle` file
before extraction. From offset 0:

```
offset           size       field                description
---------------  ---------  -------------------  ------------------------------
0                8          magic                ASCII "HEKABUND" (0x48 0x45
                                                  0x4B 0x41 0x42 0x55 0x4E 0x44)
8                4          version              big-endian u32; v1 = 0x0000_0001
12               4          manifest_len         big-endian u32; N bytes; MUST
                                                  be ≤ 8 * 1024 * 1024 (8 MiB)
16               N          manifest_json        UTF-8 JSON, see §3
16 + N           M          concat_data          per-entry file bodies in
                                                  manifest order; total M bytes
16 + N + M       32         trailer_sha256       SHA-256 over bytes
                                                  [0 .. 16 + N + M], i.e. magic
                                                  + version + manifest_len +
                                                  manifest_json + concat_data;
                                                  trailer itself excluded
```

**Total bundle size:** `52 + N + M` bytes (16-byte header + manifest + data +
32-byte trailer). The `PayloadTransferFrame.total_size` advertised in
`Introduction` MUST equal this exact value.

### 2.1 Endianness and field widths

All scalar fields (`version`, `manifest_len`, `trailer_sha256` length-context)
are encoded in **big-endian** byte order. JSON numerics inside `manifest_json`
follow standard JSON encoding (no endianness; the parser handles them).
Big-endian is chosen for cross-platform reproducibility in hex dumps,
fuzz corpora, and protocol captures — consistent with `chunk-hmac.md` §4.2.

### 2.2 Magic byte gate

The 8-byte ASCII magic `HEKABUND` is the **first** receiver-side check.
A receiver that opens a `.bundle` temp file MUST refuse to proceed if the
first 8 bytes do not match exactly; in that case the bundle is treated as
corrupted and the temp file is removed (per §6 atomic-reject). The magic
is **not** a protobuf field — it is raw bytes, scanned without proto
parsing, identical in spirit to the `0xA5DEB201` magic in
`capabilities.md` §2.

### 2.3 manifest_len cap

`manifest_len ≤ 8 MiB` (`0x0080_0000`). Larger values MUST be rejected
without parsing the manifest (DoS guard; an 8 MiB manifest comfortably
fits 10,000 entries × ~600 bytes each, see RFC-0005 §3.3). A `manifest_len`
of 0 is invalid (the manifest MUST contain at least `version`, `root_name`,
`total_entries`, and `entries`). A `manifest_len` that would push the total
header past `concat_data` start (i.e. larger than the remaining payload)
MUST be rejected.

### 2.4 concat_data layout

`concat_data` is the bytewise concatenation of every entry's body whose
`type == "file"`, in **the exact order** the entries appear in
`manifest.entries`. Directory entries contribute zero bytes. There is **no**
per-entry framing, length prefix, or padding inside `concat_data` — the
receiver tracks position by accumulating each file entry's `size` from the
manifest. The total length of `concat_data` MUST equal the sum of all
`entries[*].size` values where `type == "file"`; mismatch is a protocol
violation and triggers atomic-reject.

### 2.5 trailer_sha256

The final 32 bytes are the SHA-256 digest computed over bytes
`[0 .. bundle_len - 32]` — that is, every byte of the bundle **except** the
trailer itself. This binds magic, version, manifest, and all file data
together into a single tamper-evident commit. Receivers MUST hold back the
last 32 bytes of the streamed payload from disk verification scope and
recompute the digest after the full payload arrives (see §6). Mismatch =
atomic-reject.

## 3. Manifest JSON schema (v1)

The manifest is an in-bundle UTF-8 JSON document. **It is not represented
on the proto wire** — the `FolderManifest` protobuf message
(`HekaDropFrame.payload.folder_mft`, slot 14) is intentionally empty in v1
and serves only to seal the slot for ABI stability (see §7 and
`crates/hekadrop-proto/proto/hekadrop_extensions.proto`). The manifest
travels exclusively inside the `HEKABUND` container.

### 3.1 Top-level shape

```ts
interface BundleManifest {
  version:        1;                  // schema version; only 1 in v0.8.x
  root_name:      string;             // sanitized folder name; receiver-visible
  total_entries:  number;             // u32; MUST equal entries.length
  created_utc:    string;             // ISO-8601 UTC, e.g. "2026-04-28T12:34:56Z"
  entries:        ManifestEntry[];    // ordered; concat_data follows this order
}

type ManifestEntry = FileEntry | DirectoryEntry;

interface FileEntry {
  type:    "file";
  path:    string;                    // root-relative, POSIX `/` separator only
  size:    number;                    // u64; bytes; 0 ≤ size ≤ i64::MAX
  sha256:  string;                    // 64-char lowercase hex digest of file body
  mode?:   number;                    // optional POSIX mode (decimal, e.g. 420 = 0o644)
  mtime?:  number;                    // optional Unix epoch seconds (u64)
}

interface DirectoryEntry {
  type:    "directory";
  path:    string;                    // root-relative, POSIX `/` separator only
  mode?:   number;                    // optional POSIX mode (decimal, e.g. 493 = 0o755)
  mtime?:  number;                    // optional Unix epoch seconds (u64)
}
```

### 3.2 Field semantics (normative)

| Field | Constraint | Action on violation |
|---|---|---|
| `version` | MUST be the integer `1`; any other value reject the bundle | Atomic-reject |
| `root_name` | Non-empty string; passes per-segment sanitize (§5); MUST NOT contain `/` or `\` | Atomic-reject |
| `total_entries` | MUST equal `entries.length`; bounded above by 10 000 | Atomic-reject |
| `created_utc` | Strict ISO-8601 in `YYYY-MM-DDTHH:MM:SSZ` form (UTC, trailing Z) | Atomic-reject |
| `entries[*].type` | MUST be exactly `"file"` or `"directory"`; unknown types reject | Atomic-reject |
| `entries[*].path` | Forward-slash separator; per-segment sanitize (§5); depth ≤ 32 | Atomic-reject |
| `entries[file].size` | `0 ≤ size ≤ i64::MAX`; sum of all file sizes equals `concat_data` length | Atomic-reject |
| `entries[file].sha256` | Exactly 64 lowercase hex characters; matches streamed body | Atomic-reject |
| `entries[*].mode` | Optional u32; ignored on platforms that cannot apply it | Silently dropped (best-effort) |
| `entries[*].mtime` | Optional u64; Unix epoch seconds; ignored on best-effort platforms | Silently dropped (best-effort) |

### 3.3 Forward-compatibility rules

- Unknown **top-level** fields → ignored (parser tolerates).
- Unknown **entry-level** fields → ignored.
- Unknown `entries[*].type` value → bundle reject (cannot guess data layout).
- Schema bump (`version: 2`) MUST be gated by a new capability bit
  (`FOLDER_STREAM_V2`); a v1-only receiver receiving `version: 2` MUST reject.

### 3.4 Symlink and special-file omission

The schema defines **no** `"symlink"`, `"hardlink"`, `"socket"`, `"pipe"`,
`"block"`, or `"char"` entry types. Senders MUST skip these during folder
walk (RFC-0005 §3.4). A receiver that observes any unknown `type` value
treats it as a protocol violation and rejects the bundle.

## 4. Introduction-frame contract (MIME marker + attachment_hash)

A folder bundle is announced inside `IntroductionFrame` as a single
`FileMetadata` (RFC-0005 §3.1):

```
FileMetadata {
  name:            "<root_name>.hekabundle"      // receiver display hint
  type:            UNKNOWN                        // not IMAGE/VIDEO/AUDIO/etc.
  payload_id:      <i64>                          // standard
  size:            <bundle total bytes>           // 52 + N + M (see §2)
  mime_type:       "application/x-hekadrop-folder"
  id:              <i64 attachment_id>            // standard
  parent_folder:   ""                              // unused; folder-in-folder N/A
  attachment_hash: <i64>                           // see §4.1
}
```

### 4.1 `attachment_hash` derivation

`attachment_hash` is a **64-bit signed integer** carrying introduction-time
pre-commit integrity. It is computed as:

```
manifest_sha256 = SHA-256(manifest_json)            // 32 bytes; over JSON UTF-8
                                                     // bytes only, BEFORE bundle wrap
prefix_8        = manifest_sha256[0..8]              // first 8 bytes, big-endian
attachment_hash = i64::from_be_bytes(prefix_8)       // signed; sign bit propagates
```

The receiver, after parsing `manifest_json` from the bundle, recomputes
`manifest_sha256` and compares the first-8-byte i64 reading against the
introduction-time `attachment_hash`. **Mismatch = atomic-reject.** This
catches a tampered manifest where the trailer SHA was also recomputed (a
corrupted-sender or MITM scenario where attacker controls everything except
the introduction frame's signed envelope).

### 4.2 MIME marker contract

`mime_type == "application/x-hekadrop-folder"` is the **sole** signal a
receiver uses to decide bundle-extraction vs. raw-file save. The string
match is exact (case-sensitive per RFC 6838). A peer that has not
negotiated `FOLDER_STREAM_V1` MUST still treat this MIME as opaque and
save the file as-is — but in practice the sender will not transmit a
folder bundle to a peer lacking the capability (§7 fallback). Defensive
note: receivers SHOULD log `info` when receiving this MIME without
`FOLDER_STREAM_V1` active, since it indicates a sender bug or downgrade
attack.

### 4.3 Receiver dispatch

When the introduction lands and capability check passes, the receiver:

1. Allocates `~/Downloads/.hekadrop-temp-<session_id>.bundle` via the
   existing `unique_downloads_path` + `OpenOptions::create_new(true)`
   guard (no race with concurrent transfers).
2. Registers this path as the `PayloadAssembler` destination for the
   payload_id (existing `register_file_destination` API).
3. After payload completion, transitions to the §6 atomic-reject state
   machine.

## 5. Path sanitization policy

Folder bundles introduce **per-segment** path sanitization on top of the
existing basename-only `sanitize_received_name` helper. Each manifest entry's
`path` is processed as:

1. **Separator check.** If `path` contains `\` (Windows backslash), reject
   the bundle. Senders are required to POSIX-normalize paths during walk;
   any backslash on the wire indicates a non-conformant sender.
2. **Split.** `segments = path.split('/').collect::<Vec<_>>()`.
3. **Per-segment sanitize.** Each segment is fed through the existing
   `sanitize_received_name` rules (NUL/control reject, Windows reserved
   names, length ≤ 200 bytes, leading/trailing dot collapse).
4. **Special-segment handling.**
   - `""` (empty, e.g. from leading `/` or `//`): reject.
   - `"."`: skipped silently.
   - `".."`: **reject the entire bundle** (path traversal). Note: the
     basename-only `sanitize_received_name` rewrites `".."` to `"dosya"`,
     which is fine for one-shot files but unsafe in a multi-segment
     context (silent path mutation could collide entries).
5. **Empty result.** If sanitization removes all segments, reject.
6. **Depth cap.** `segments.len() > 32` → reject (DoS / pathological-nesting
   guard, RFC-0005 §3.3).
7. **Total length cap.** No explicit limit beyond the per-segment 200-byte
   cap; a 32-deep path of 200-byte segments stays well below filesystem
   PATH_MAX (typically 4096 on Linux, 1024 on macOS, 260 on Windows
   without long-path opt-in — long-path support TBD v0.9).

### 5.1 Cross-platform path normalization

- **Sender side.** During folder walk, the sender MUST convert OS-native
  separators to `/` before writing manifest paths. On Unix this is a no-op;
  on Windows the sender replaces `\` with `/`. Drive letters and UNC
  prefixes (e.g. `C:\`, `\\server\share\`) MUST be stripped — the manifest
  carries only the *root-relative* path under the user-selected folder.
- **Receiver side.** The receiver always processes paths as `/`-separated
  in the manifest; it is the receiver's `Path` builder that converts to
  the OS-native separator when creating files in
  `~/Downloads/.hekadrop-temp-<session>/`. On Windows this means joining
  segments with `\`; on Unix with `/`.

### 5.2 Symlink TOCTOU guard

Before opening any extracted file for write, the receiver MUST
`symlink_metadata` the parent directory and verify it is a **plain
directory** (not a symlink). This prevents a TOCTOU race where the parent
becomes a symlink between `create_dir_all` and the file open — the existing
`payload.rs:349-360` pattern generalizes here. Failure → atomic-reject.

## 6. Atomic-reject receive state machine

The receiver state machine is **single-commit**: either every entry
materializes in `~/Downloads/<root_name>/` or nothing does. There is no
partial-accept mode (RFC-0005 §3.6 closed open question 6). The sequence:

```
   ┌────────────────────────────────────────────────────────┐
   │ 1. payload chunks → write to                           │
   │    ~/Downloads/.hekadrop-temp-<session>.bundle         │
   └────────────────────────────────────────────────────────┘
                            │
                            ▼
   ┌────────────────────────────────────────────────────────┐
   │ 2. payload complete:                                   │
   │    - read trailer (last 32 bytes)                      │
   │    - SHA-256 of [0..bundle_len-32]                     │
   │    - constant-time compare                             │
   └────────────────────────────────────────────────────────┘
                            │  match
                            ▼
   ┌────────────────────────────────────────────────────────┐
   │ 3. parse manifest_json:                                │
   │    - magic check                                       │
   │    - version == 1                                      │
   │    - manifest_len ≤ 8 MiB                              │
   │    - JSON parse → BundleManifest                       │
   │    - schema validate (§3.2)                            │
   │    - SHA-256(manifest_json) prefix vs. attachment_hash │
   └────────────────────────────────────────────────────────┘
                            │  ok
                            ▼
   ┌────────────────────────────────────────────────────────┐
   │ 4. sanitize root_name → safe_root                      │
   │    mkdir ~/Downloads/.hekadrop-temp-<session>/         │
   │    (create_new; session-id unique → no EEXIST race)    │
   └────────────────────────────────────────────────────────┘
                            │
                            ▼
   ┌────────────────────────────────────────────────────────┐
   │ 5. for entry in manifest.entries (in order):           │
   │      sanitize entry.path → segments (§5)               │
   │      switch entry.type:                                │
   │        directory → create_dir_all (joined path)        │
   │        file      → open create_new                     │
   │                    parent symlink check                │
   │                    copy entry.size bytes from bundle   │
   │                    streaming SHA-256 update            │
   │                    on EOF: compare to entry.sha256     │
   │                                                         │
   │      ANY failure → goto REJECT                          │
   └────────────────────────────────────────────────────────┘
                            │  all entries ok
                            ▼
   ┌────────────────────────────────────────────────────────┐
   │ 6. unique_downloads_path(safe_root)                    │
   │    rename(.hekadrop-temp-<session>/, target/)          │
   │      EXDEV → fallback recursive copy + delete          │
   │    delete .hekadrop-temp-<session>.bundle              │
   │    UI: "klasör alındı: <safe_root> (N dosya)"          │
   └────────────────────────────────────────────────────────┘

   REJECT path (any failure above):
   ┌────────────────────────────────────────────────────────┐
   │ - rm -rf ~/Downloads/.hekadrop-temp-<session>/         │
   │ - rm ~/Downloads/.hekadrop-temp-<session>.bundle       │
   │ - log + UI: "bundle reddedildi: <neden>"               │
   │ - stats.json: abort kaydı                              │
   │ - send Disconnection frame to sender                   │
   └────────────────────────────────────────────────────────┘
```

**No partial materialization** ever reaches `~/Downloads/<root_name>/`.
The temp directory is the staging area; success = atomic `rename`,
failure = recursive delete.

## 7. Capabilities gate

`FOLDER_STREAM_V1` occupies bit `0x0004` of `Capabilities.features`
(RFC-0005 §3.7). The capability table shared across RFCs 0003/0004/0005:

```
CHUNK_HMAC_V1     = 0x0001    (RFC-0003, bit 0)
RESUME_V1         = 0x0002    (RFC-0004, bit 1)
FOLDER_STREAM_V1  = 0x0004    (RFC-0005, bit 2)
0x0008 .. 0x8000  reserved for v0.8 – v1.0
```

**Negotiation flow** (per `capabilities.md` §3):

1. Both peers send `Capabilities{version=1, features=...}` after
   `PairedKeyEncryption`.
2. `active_caps = my_features & peer_features`.
3. If `active_caps & FOLDER_STREAM_V1 == 0`, sender takes the **fallback
   path**: builds a flat multi-file `IntroductionFrame` via the existing
   `build_introduction_multi` flow (RFC-0005 §3.7), and the UI surfaces a
   discrete warning ("Bu alıcıda klasör yapısı korunmayacak.").
4. If `active_caps & FOLDER_STREAM_V1 != 0`, sender constructs the
   `HEKABUND` bundle in memory and sends it as a single `FILE` payload
   with the §4 introduction.

**Receiver-side defensive behavior:** A receiver that has not advertised
`FOLDER_STREAM_V1` but receives a `FileMetadata` with
`mime_type == "application/x-hekadrop-folder"` MUST save the bundle as a
raw `.hekabundle` file in `~/Downloads/` (do **not** attempt extraction —
the receiver lacks the negotiated capability). Log at `warn` level; this
indicates a sender bug or downgrade attack.

## 8. Failure modes & recovery

| Trigger | Receiver action | Sender action | Resume behavior |
|---|---|---|---|
| Magic mismatch (`bundle[0..8] != "HEKABUND"`) | Atomic-reject; remove `.bundle`; send Disconnection | Reconnect with `offset = 0`; treat partial state as untrusted | None |
| `version != 1` | Atomic-reject; UI: "desteklenmeyen bundle sürümü" | Sender bug — should have negotiated FOLDER_STREAM_V<n> | None |
| `manifest_len > 8 MiB` | Atomic-reject before parsing manifest (DoS guard) | Sender bug | None |
| `manifest_len == 0` or overflows past payload end | Atomic-reject | Sender bug | None |
| trailer SHA-256 mismatch | Atomic-reject; remove temp+`.bundle`; UI: "bundle integrity failed" | Reconnect with `offset = 0` | None — full restart |
| manifest JSON parse fail (malformed UTF-8, syntax error) | Atomic-reject; UI: "bundle manifest okunamadı" | Sender bug | None |
| `attachment_hash` prefix mismatch (manifest tampered post-introduction) | Atomic-reject; UI: "manifest commit mismatch" | Sender bug or active MITM | None |
| Schema validation fail (`type` unknown, `total_entries` mismatch, depth > 32, etc.) | Atomic-reject with §3.2 reason | Sender bug | None |
| Path sanitize fail (`..`, backslash, empty, oversized) | Atomic-reject; UI includes failing path (redacted via `path_basename`) | Sender bug or hostile | None |
| Per-file SHA-256 mismatch | Atomic-reject; remove temp+`.bundle`; UI: "dosya `<path>` integrity failed" | Reconnect with `offset = 0` | None — atomic-reject (RFC-0005 §3.6) |
| `concat_data` length mismatch (sum of file sizes ≠ payload remainder) | Atomic-reject | Sender bug | None |
| Disk full during extract | Atomic-reject; UI: "disk dolu, klasör alınamadı" | None (extract is local) | None — user must free space |
| `mkdir` of temp directory fails | Atomic-reject; UI: "geçici klasör oluşturulamadı" | None | None |
| Parent-symlink race detected (§5.2) | Atomic-reject; UI: "filesystem race detected" | None | None |
| `rename` fails (cross-device EXDEV) | Fallback: recursive copy then delete temp; if copy also fails → atomic-reject | None | None |

**Cleanup invariant:** On every reject path, both
`.hekadrop-temp-<session>/` (the directory, recursive) and
`.hekadrop-temp-<session>.bundle` (the file) MUST be removed before
returning the error to the UI layer.

## 9. Test vectors (KAT)

These known-answer tests are **provisional placeholders** until the wire
format is exercised end-to-end during PR-B (HEKABUND encoder/decoder
primitives) and PR-D (receiver extract pipeline). Once finalized, the
canonical bytes will be committed under
`docs/protocol/captures/folder-payload-kat-*.bin` along with annotated
`.md` walkthroughs.

### KAT-1 — 2-file folder roundtrip

```
root_name        = "kat1"
created_utc      = "2026-01-01T00:00:00Z"
entries          = [
  { type: "file", path: "a.txt", size: 5, sha256: <hex of "hello"> },
  { type: "file", path: "b.txt", size: 5, sha256: <hex of "world"> },
]

manifest_json    = (canonical JSON; field order as §3.1; no whitespace)
manifest_len     = len(manifest_json)
concat_data      = b"hello" || b"world"   (10 bytes total)

bundle           = "HEKABUND"
                || 0x00 0x00 0x00 0x01           (version)
                || manifest_len.to_be_bytes()
                || manifest_json
                || concat_data
                || trailer_sha256                 (32 bytes)

attachment_hash  = i64::from_be_bytes(SHA-256(manifest_json)[0..8])

Expected receiver outcome:
  ~/Downloads/kat1/a.txt   contents "hello"
  ~/Downloads/kat1/b.txt   contents "world"
  no .hekadrop-temp-* artifacts remaining
```

### KAT-2 — Empty folder with one directory entry

```
root_name        = "kat2"
created_utc      = "2026-01-01T00:00:00Z"
entries          = [
  { type: "directory", path: "subdir" },
]

manifest_len     = len(manifest_json)
concat_data      = (empty; 0 bytes)

bundle           = "HEKABUND"
                || 0x00 0x00 0x00 0x01
                || manifest_len.to_be_bytes()
                || manifest_json
                || (no concat_data)
                || trailer_sha256

Expected receiver outcome:
  ~/Downloads/kat2/        (directory exists)
  ~/Downloads/kat2/subdir/ (directory exists, empty)
  no .hekadrop-temp-* artifacts remaining
```

### KAT-3 — Bundle with pathological manifest (negative test, planned)

A bundle whose manifest contains `path: "../escape"` MUST be rejected at
step 5 of the §6 state machine; no temp directory entries are created
beyond the initial `mkdir`. Used by `fuzz_bundle_parser` as a regression
seed.

The full KAT corpus will land in `docs/protocol/captures/` during PR-B/PR-D
implementation; the directory does not yet exist.

## 10. Security considerations

| Vector | Mitigation |
|---|---|
| Path traversal (`../etc/passwd`) | Per-segment sanitize (§5); `..` rejects entire bundle, not silent rewrite |
| Absolute path (`/etc/passwd`, `C:\Windows\…`) | Leading `/` produces empty first segment → reject; backslash → reject (§5) |
| Symlink attack on parent directory | Pre-write `symlink_metadata` parent check (§5.2); generalizes `payload.rs:349-360` |
| Zip-slip equivalent | Bundle paths root-relative; `create_new` open flag + temp-dir staging |
| Manifest JSON injection (oversized, malformed) | `manifest_len ≤ 8 MiB` cap (§2.3); strict schema (§3.2); unknown `type` rejects |
| Resource exhaustion (entry count) | 10 000 entry cap (RFC-0005 §3.3) |
| Resource exhaustion (depth) | 32-segment depth cap (§5) |
| Bundle tampering (data swap, manifest swap) | Trailer SHA-256 (§2.5) + per-file SHA-256 (§6 step 5) + introduction-time `attachment_hash` commit (§4.1) |
| Partial bundle poisoning | Atomic-reject (§6); no entries reach `~/Downloads/` until every entry verified |
| Disk-fill DoS (hostile sender) | **Open question deferred to v0.8.1**: pre-flight `available_space` check (RFC-0005 §11.5) |
| TOCTOU on `rename` | `rename` is atomic on same FS; cross-device fallback uses `create_dir_all` + `create_new` per file |
| Downgrade attack (sender lies about `FOLDER_STREAM_V1`) | Receiver-side defensive log (§7); raw save instead of extraction |

**No new crypto primitives** are introduced. SHA-256 is reused from the
existing `hekadrop-core::crypto` module; no key material is added (the
trailer is a plain digest, not a MAC — the bundle's authenticity comes
from `SecureCtx` at the wire layer plus `attachment_hash` commit at
introduction time).

The threat-model document (`docs/security/threat-model.md` §7.1 path
traversal, §7.3 symlink race) will be updated during PR-D to record the
folder-bundle generalization of these mitigations.
